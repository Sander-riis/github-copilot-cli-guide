CLI Overview

GitHub Copilot CLI is a standalone terminal AI agent — not a wrapper around gh and not the same as the IDE extension. You invoke it with copilot from any shell, and it maintains its own session context, tool permissions, and execution mode stack. Unlike the IDE extension, the CLI can traverse your filesystem, run shell commands, create and modify files, and interact with GitHub — without switching context from your terminal.

The CLI is built for developers who work in the terminal. Run it interactively for open-ended exploration, send focused one-shot prompts with -p, or wire it into scripts. The same agents and skills ecosystem available in Copilot cloud works here too — your custom .agent.md files and .github/skills/ directory are picked up automatically.

If you're new to Copilot CLI, don't start with the full reference. Run one short loop that shows how the tool behaves in practice.

Command Patterns

Four core invocation forms cover the majority of use cases. The bare copilot command opens a persistent interactive session; -p sends a single prompt and exits.

bash 
# Interactive session — opens a persistent context window
copilot

# Non-interactive with a prompt — responds and exits
copilot -p "refactor src/auth/login.ts to use async/await"

# Short alias for -p
copilot -sp "what does the payment module do?"

# Launch with a specific custom agent active from the start
copilot --agent code-reviewer "review the auth module"
Mode Cycling — Standard, Plan, Autopilot

During any interactive session, press Shift+Tab to cycle through three execution modes. The active mode is shown in the session prompt. You can also set the starting mode from the command line before the session opens.

ModeBehaviorBest For
Standard Normal interactive — asks for confirmation before each tool use Exploratory tasks, anything touching production code or external systems
Plan Drafts a written implementation plan first; you review and approve before any execution begins Complex multi-file changes where you want to audit the approach before committing
Autopilot Auto-confirms all tool requests — fully autonomous execution Trusted scoped tasks; CI-adjacent workflows; combined with /fleet
Tip

Start a session directly in autopilot mode with the --autopilot flag: copilot --autopilot -p "run the full test suite and fix any failures"

Tool Access & Permissions

The CLI agent's effectiveness depends entirely on which tools it can use. Without tool access, Copilot can only suggest — it cannot read your code, run commands, or make changes. Every meaningful action flows through the tool permission system.

Available Tools

Copilot CLI has four core tools. Each one unlocks a different category of work:

ToolWhat It EnablesWithout It
viewRead files, list directories, inspect project structureAgent cannot see your code — answers are based on guesswork
shellRun terminal commands (build, test, lint, git, package managers)Agent cannot verify its changes work, run tests, or install dependencies
writeCreate and edit filesAgent can only suggest changes — you must apply them manually
searchGrep, glob, and code search across the codebaseAgent cannot find symbols, trace imports, or locate usages

The Permission Model

By default, Copilot asks for confirmation before each tool use. This is safe but slow — especially for multi-step tasks where the agent needs dozens of tool calls. The three execution modes control how permissions flow:

ModeBehaviorBest For
Interactive (default)Asks permission for every tool callUnfamiliar tasks, learning what Copilot does
Plan (Shift+Tab)Agent plans but doesn't execute until you approveComplex changes where you want to review the approach first
Autopilot (Shift+Tab ×2)Agent executes freely — no confirmationsTrusted tasks, scripted pipelines, bulk operations
Key Insight

Denying tool access doesn't make Copilot safer — it makes it less useful. If you block shell, the agent can write code but can't verify it compiles. If you block view, it can't read your existing code and will hallucinate file structures. Grant the tools the task requires, and use plan mode if you want oversight without sacrificing capability.

Fine-Grained Control

For cases where you want selective permissions, use startup flags to pre-approve or deny specific tools:

bash 
# Allow all tools — fully autonomous
copilot --allow-all-tools -p "refactor the auth module"

# Allow everything (tools + paths + URLs)
copilot --yolo -p "set up the CI pipeline"

# Pre-approve a specific tool pattern
copilot --allow-tool='shell(npm test)' -p "fix the failing tests"

# Block write access — read-only audit
copilot --deny-tool='write' -p "review src/ for security issues"

For custom agents, use the tools: field in the agent's YAML frontmatter to permanently restrict what an agent can do — regardless of session-level permissions. An agent with only [view, search] can never modify files, even in autopilot mode.

Best Practice Workflows

Most day-to-day work fits one of three loops. Pick the smallest one that matches the task instead of reaching for every feature at once.

Complex change — explore → plan → implement

Use this for multi-file features, refactors, or unfamiliar codebases. The goal is to make the agent show its understanding before it starts editing.

text 
Read the auth files but don't change anything yet
/plan Add password reset to the API
Review the plan, tweak it, then say "Proceed with the plan"
Small fix — describe → patch → verify

Skip plan mode for tight, local changes. Be explicit about scope so the agent doesn't widen the task on its own.

text 
Fix the null check in src/api/users.ts only
Run the relevant tests
Commit with a clear message when you're done
Before merge — review → revise → ship

Use the built-in review path before you open a PR or merge a risky change. This catches logic and security issues without turning every run into a full planning session.

text 
/review against main — focus on bugs and security
Apply the fixes
Open the PR or commit once the review comes back clean

Agents Overview

Info

Two agent systems share the same directory. Copilot CLI agents use the .agent.md double extension and are invoked locally from your terminal. GitHub cloud agents (the Copilot coding agent on GitHub.com Issues and PRs) use plain .md files. Both live in .github/agents/ — the file extension is the only visual difference. This guide covers CLI agents only.

A Copilot CLI agent is a persona with its own isolated context window, a defined set of tool permissions, and a system prompt that shapes its behavior. When you invoke an agent, Copilot doesn't continue the current session — it opens a separate context and routes your request through the agent's instructions. This means agents can be highly specialized: a security auditor that never writes files, a test runner that only touches test directories, or a deployment agent that follows a strict rollout checklist.

Agents differ from copilot-instructions.md custom instructions in that they're invokable on demand and session-scoped, rather than always-on. You can have many agents in your repo and invoke only the one suited to the current task. Agents can also be invoked by other agents as subagents — enabling parallel, coordinated workflows.

Agent Profile Format

A CLI agent is a file with the .agent.md double extension, placed in .github/agents/. The file has two parts:

  1. YAML frontmatter — defines identity, tool permissions, and MCP connections
  2. Markdown body — becomes the agent's system prompt (the instructions it follows)

Only description is required; all other frontmatter fields are optional.

.github/agents/code-reviewer.agent.md 
---
name: code-reviewer
description: >
  Reviews code for bugs, security issues, and style violations.
  Use when asked to review code, audit a PR, or check for vulnerabilities.
tools:
  - shell
  - view
  - search
mcp-servers:
  - name: github
    type: http
    url: https://api.githubcopilot.com/mcp/
---

# Code Reviewer Agent

You are a code review agent. Your role is to analyze code changes
for bugs, security vulnerabilities, performance issues, and
style violations.

## Core Principles

- Read the full diff before commenting on individual files
- Prioritize correctness and security over style nits
- Provide actionable suggestions with code examples
- Flag blocking issues separately from minor improvements
Anatomy

The --- fences wrap YAML frontmatter. Everything after the closing --- is the Markdown system prompt — write it like you're briefing a colleague. The description field is what Copilot reads to decide when to auto-invoke this agent, so be specific.

Frontmatter Field Reference
FieldRequired?TypePurpose
name Optional string Human-readable display name. If omitted, the filename without .agent.md is used as the agent ID for invocation (e.g., code-reviewer.agent.md → invoked as code-reviewer).
description Required string Tells Copilot when to invoke this agent. Used for trigger-word inference and prompt-mention matching. Be specific — vague descriptions reduce auto-routing accuracy. This is the most important field.
tools Optional list of strings Restricts the agent to a specific tool subset. Valid values: shell, view, write, search. If omitted, all available tools are granted. Use this to create read-only agents or agents with limited filesystem access.
mcp-servers Optional list of objects Attaches MCP (Model Context Protocol) servers to this agent's context. Each entry requires name, type, and url. The GitHub MCP server (https://api.githubcopilot.com/mcp/) gives the agent access to GitHub API operations.
Scope

Custom agents can live in .github/agents/ (repo), ~/.copilot/agents/ (personal), or .github-private/agents/ (org). If the same name exists in multiple places, personal overrides repo, and repo overrides org.

Invocation Methods

There are four ways to invoke a custom agent. They can be combined — the --agent flag starts the session with a specific agent pre-selected, and then /agent can switch to a different one mid-session.

#MethodExampleWhen to Use
1 Slash command /agent code-reviewer Switch the active agent explicitly mid-session
2 Prompt mention "Use the security-auditor agent to review this PR" Natural-language invocation — Copilot resolves the agent by matching the name in your prompt
3 Trigger-word inference "audit src/ for injection vulnerabilities" Copilot reads each agent's description: field and auto-routes based on keywords — no explicit agent name required

/agent code-reviewer

# Method 4: start session with agent pre-selected (CLI flag)
copilot --agent code-reviewer "review the auth module"

# Combine --agent + --autopilot for non-interactive scripted workflows
copilot --agent code-reviewer --autopilot -p "review all changed files"

Subagent Orchestration

An agent becomes an orchestrator when it invokes other agents as subagents. The orchestrating agent receives the full task, decomposes it, assigns each piece to the appropriate subagent, dispatches them, and synthesizes the results. Each subagent runs in its own isolated context window — it does not share memory with the orchestrator or with other subagents running simultaneously.

Context isolation is the architectural property that makes parallelism safe. A code-review subagent and a task subagent can both inspect the same codebase simultaneously without their findings interfering. The orchestrator sees only the subagent outputs, not the intermediate reasoning each accumulated. This also means subagents cannot accidentally leak sensitive context between tasks.

Fleet Execution — /fleet

/fleet is the high-level command for parallel subagent execution. When you type /fleet "task description", Copilot acts as the orchestrator: it analyzes the task, identifies parallelizable subtasks, dispatches each to a subagent with an isolated context window, and synthesizes the combined output into a coherent result.

Fleet works best combined with autopilot mode — without it, each subagent stops to ask for tool confirmations, eliminating the speed benefit of parallelism. Switch to Autopilot with Shift+Tab before issuing /fleet, or start the session with --autopilot.

bash 
# Start in autopilot mode first, then fleet a task
copilot --autopilot
# (session opens in autopilot mode)
/fleet "add input validation to all API endpoints in src/routes/"

# Or switch to autopilot mid-session before using /fleet
# Press Shift+Tab until prompt shows [autopilot], then:
/fleet "write unit tests for every exported function in src/utils/"
Warning

/fleet is a preview feature. It is functional in current releases but its syntax and parallel execution behavior may change. Check /help in your active session to verify the current interface before building workflows that depend on it.

Agentic Code Review — /review

The /review command invokes the code-review agent to analyze your changes without leaving the CLI. It only surfaces genuine issues — bugs, security vulnerabilities, logic errors — and never comments on style or formatting.

  1. Type /review and optionally specify a scope (path, file pattern, or prompt)
  2. If Copilot proposes running a command (e.g., inspecting a diff), review and approve it
  3. Read the feedback and apply improvements in your editor
bash 
# Review all uncommitted changes
/review

# Scoped review with multi-model comparison
/review Use Opus 4.5 and Codex 5.2 to review changes against main. Focus on bugs and security.
Steering Agents While Thinking

While Copilot is working on a task, you can send messages at any time to steer its behavior — no need to wait for it to finish. Input sent while Copilot is thinking is processed in order as part of the active task.

Use steering to:

  • Interrupt an agent heading in the wrong direction
  • Provide inline feedback when rejecting a tool permission request
  • Refine scope or clarify requirements mid-execution
Tip

Use Ctrl+Enter or Ctrl+Q to queue a message while the agent is busy. The message will be processed when the agent is ready.

Skills Overview

bash 
.github/
└── skills/
    └── my-skill/
        ├── SKILL.md          # required — skill definition

Skills are reusable instruction bundles — unlike agents, they don't get their own context window. When you invoke a skill with /skill-name, Copilot loads the skill's instructions into the current session alongside your conversation. Think of skills as modular expertise you can pull in on demand: a deploy checklist, a code review procedure, a database migration workflow. They're lighter than agents (no isolated context, no persona) but more structured than a raw prompt.

Info

Skills are an open standard. The Agent Skills specification defines a portable format that works across Copilot CLI, GitHub cloud agents, and VS Code agent mode. A SKILL.md file you write for the CLI works identically in all three surfaces — no adaptation needed.

SKILL.md Format

A skill is defined by a SKILL.md file inside a named subdirectory under .github/skills/. The directory name becomes the skill's invocation name. YAML frontmatter declares the skill's identity and permissions; the Markdown body becomes the instructions Copilot follows when the skill is active.

.github/skills/deploy/SKILL.md 
---
name: deploy                              # required — lowercase-hyphens; invoked as /deploy
description: >                            # required — Copilot reads this to decide when to suggest
  Deploys the current project to the configured environment. Reads deployment
  config from deploy.yaml, runs pre-deploy checks, and executes the deploy
  pipeline. Use when asked to deploy, release, or push to production.
license: MIT                              # optional — metadata only; not enforced by Copilot
allowed-tools:                            # optional — restricts which tools this skill may use
  - shell                                 #   ⚠ grants arbitrary command execution
  - view                                  #   read files and directories
---

# Deploy Skill

When invoked, follow this sequence:

1. Read `deploy.yaml` from the project root
2. Run pre-deploy checks: `npm test` and `npm run lint`
3. If checks pass, execute the deploy command from the config
4. Report the deployment status with the environment URL
Frontmatter Field Reference
FieldRequired?TypePurpose
name Yes string (lowercase-hyphens) Skill identifier — this is what you type after / to invoke the skill (e.g., /deploy). Must match the directory name.
description Yes string Tells Copilot what this skill does. Used for suggestion matching — when a prompt aligns with the description, Copilot may recommend the skill. Be specific about the task domain.
license No string License identifier (e.g., "MIT"). Metadata only — not enforced by Copilot. Useful for shared or community skills.
allowed-tools No list of strings Tools this skill may use. Valid values: shell, view, write, search. If omitted, the skill inherits the session's tool permissions. shell grants arbitrary command execution — see security warning below.
Script-Enabled Skill Example

A skill can include companion scripts that Copilot executes via allowed-tools: shell. The SKILL.md body references the script by relative path; Copilot runs it when the skill is invoked.

.github/skills/db-migrate/SKILL.md 
---
name: db-migrate
description: >
  Runs database migrations using the companion shell script. Use when asked
  to migrate, update schema, or apply pending database changes.
allowed-tools:
  - shell
---

# Database Migration Skill

Run the companion migration script located at `./helpers/migrate.sh`.
Pass the target environment as the first argument.
.github/skills/db-migrate/helpers/migrate.sh 
#!/usr/bin/env bash
set -euo pipefail

ENV="${1:-development}"
echo "Running migrations for: $ENV"
npx prisma migrate deploy --schema=./prisma/schema.prisma

Danger

allowed-tools: shell grants arbitrary command execution. A skill with shell access can run any command on your machine with your user permissions — including modifying files, installing packages, accessing credentials, and making network requests. Only enable shell access for skills you have personally reviewed and trust. Never grant shell access to third-party skills without reading every line of their source.

Scope

Skills usually live in .github/skills/ when you want them shared with the repo, or ~/.copilot/skills/ when you want them available across all of your local projects.

MCP Overview

# Add an MCP server interactively
/mcp add

# Show all registered MCP servers
/mcp show

# Delete an MCP server
/mcp delete server-name

# Reload all MCP servers
/mcp reload

The Model Context Protocol (MCP) is an open standard that lets Copilot CLI connect to external tools and data sources through lightweight "servers." Think of MCP servers as plugins that give Copilot new abilities — reading files, querying databases, calling APIs, managing cloud resources — all through a standardised interface.

How MCP works

MCP follows a client-server pattern:

  1. Copilot CLI (client) discovers available MCP servers from your configuration
  2. MCP server advertises its tools (functions it can perform)
  3. When Copilot decides a tool would help answer your question, it calls the tool through the MCP server
  4. The server returns results back to Copilot to incorporate into its response
Key concept

MCP servers run locally on your machine. They don't send data to third-party services unless the specific server is designed to (e.g., a GitHub API server). You control which servers are active and what directories/resources they can access.

Why use MCP servers?

  • Extended reach — Access databases, APIs, file systems, and cloud services directly from your Copilot conversation
  • Context injection — Let Copilot read project-specific data (Jira tickets, Confluence docs, Slack threads) without copy-pasting
  • Tool augmentation — Add specialised tools like image processing, data analysis, or deployment automation
  • Standardised protocol — Any MCP-compatible server works with any MCP-compatible client (not just Copilot)

Configuration scope

MCP servers can be configured at multiple levels:

ScopeConfig fileApplies to
Project.github/copilot/mcp.jsonEveryone working on this repo
User~/.copilot/mcp-config.jsonAll your projects on this machine
Session-only--additional-mcp-config=JSONCurrent session only (CLI flag)

Project-level config is committed to the repo so the entire team shares the same MCP servers. User-level config is for personal tools you want available everywhere.

/mcp add Add a new MCP server interactively. Fill in details with Tab, save with Ctrl+S.
/mcp show List all registered MCP servers and their status.
/mcp edit Edit an existing MCP server's configuration.
/mcp delete <name> Remove an MCP server by name.
/mcp disable <name> Temporarily disable a server without removing it.
/mcp enable <name> Re-enable a previously disabled server.
/mcp auth <name> Manage authentication for an MCP server.
/mcp reload Reload all MCP server configurations.

mcp.json format

The configuration file defines available servers. Each server entry specifies how to start the server process:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-filesystem", "/home/user/project"],
      "env": {}
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://localhost:5432/mydb"
      }
    }
  }
}
Configuration fields explained
FieldTypeDescription
commandstringThe executable to run (e.g., npx, node, python)
argsstring[]Arguments passed to the command
envobjectEnvironment variables set for this server process

Common MCP server categories

Add MCP servers in ~/.copilot/mcp-config.json (user) or .github/copilot/mcp.json (project), or use /mcp add interactively. In practice, most teams only need one or two servers to start.

CategoryUse It ForGood First Pick
FilesystemReading and writing files outside the agent's built-in tools@anthropic/mcp-server-filesystem
GitHubIssues, pull requests, code search, and repository metadata@modelcontextprotocol/server-github
DatabaseSchema inspection and safe queries against development data@modelcontextprotocol/server-postgres
Fetch / WebPulling docs, web pages, and reference material into the session@anthropic/mcp-server-fetch
Recommendation

Start with read-heavy servers first. Filesystem, GitHub, and Fetch usually give the most value with the least operational risk.

Building custom MCP servers

You can create your own MCP servers for project-specific tooling. The protocol is language-agnostic — any process that speaks JSON-RPC over stdio can be an MCP server.

Minimal custom server example (Node.js) 
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "my-tools", version: "1.0.0" });

server.tool("lookup-ticket",
  { ticket_id: z.string() },
  async ({ ticket_id }) => ({
    content: [{ type: "text", text: `Ticket ${ticket_id}: ...` }]
  })
);

const transport = new StdioServerTransport();
await server.connect(transport);

Install with npm install @modelcontextprotocol/sdk zod, then add to ~/.copilot/mcp-config.json or use /mcp add:

{
  "mcpServers": {
    "my-tools": {
      "command": "node",
      "args": ["my_mcp_server.js"]
    }
  }
}

Troubleshooting

Common issues and fixes
SymptomLikely causeFix
"Server not found" Server not in mcp.json or wrong scope Run /mcp show and verify config file location
"Connection refused" Server process failed to start Run the command manually to check for errors (e.g., missing deps)
Tools not appearing Server started but tools not registered Verify the server's tool definitions and restart Copilot CLI
Env vars not set Missing env field in config Add required env vars to the server's env object in mcp.json
